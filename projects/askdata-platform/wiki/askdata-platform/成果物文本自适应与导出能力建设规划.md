## Context

项目已完成 Pretext 文本自适应工具链的 Phase 1-2 建设（useTextFit/TextFit/ShrinkwrapBox/AdaptiveText），并已集成到看板和报告组件中。剩余两项待完成：

1. **对话气泡布局层自适应**（原 P2）：文本内容自适应已通过 SafeMarkdown `adaptive` prop 覆盖，布局层仍有优化空间
2. **导出能力扩展**（原 P3）：三条独立管线——Canvas 报告渲染器、后端 PDF CJK 支持、看板截图导出

## 1. 对话气泡布局层自适应

### 当前状态

对话气泡在 `QueryAnalysisChat.tsx` 中渲染，结构为：

```
<div className="p-4 rounded-lg bg-white border border-gray-200">
  <SafeMarkdown text={...} adaptive />
</div>
```

文本自适应已通过 SafeMarkdown `adaptive` prop 启用（Pretext 预测量 + 单步 overflow 收敛）。布局层未做优化。

### 待优化项

| 项目 | 优先级 | 现状 | 改进目标 |
|------|--------|------|----------|
| 气泡 padding 响应式 | P2 | 固定 `p-4` (16px) | 窄容器时缩减为 `p-3` 或 `p-2` |
| 代码块自适应宽度 | P2 | 固定 `text-xs` + `overflow-x-auto` | 用 ShrinkwrapBox 或 TextFit 对 SQL 代码块自适应字号 |
| 气泡 max-width | P2 | 无显式 max-width（受面板约束） | 极端宽屏下的阅读优化 max-width |

### 改进方案

#### 1.1 响应式 Padding

用 `useTextFit` 的衍生模式或纯 CSS `container queries`：

```css
/* CSS Container Query 方案（无需 JS） */
.bubble-container {
  container-type: inline-size;
}
@container (max-width: 480px) {
  .bubble-inner { padding: 12px; }
}
@container (max-width: 360px) {
  .bubble-inner { padding: 8px; }
}
```

#### 1.2 代码块自适应

在 `SafeMarkdown` 的代码块渲染中应用 `TextFit`。react-markdown 支持自定义 `components`：

```tsx
<ReactMarkdown
  components={{
    code: ({ className, children }) => (
      <TextFit maxLines={1} minFontSize={9} maxFontSize={13}>
        {String(children)}
      </TextFit>
    ),
  }}
/>
```

#### 1.3 宽屏阅读优化

在气泡容器上设置 `max-w-prose`（Tailwind prose 默认 65ch）或 `max-w-4xl`。

### 涉及文件

- `web/src/features/query-space/pages/QueryAnalysisChat.tsx` — 气泡容器 CSS
- `web/src/components/SafeMarkdown.tsx` — 代码块自定义渲染
- `web/src/components/TextFit.tsx` — 已实现

## 2. 导出能力扩展

### 2.1 Canvas 报告渲染器

#### 目标

在浏览器端使用 OffscreenCanvas + Pretext 测量 + Canvas 2D API，将报告内容（Markdown → AST → Canvas 布局）渲染为位图，支持导出 PNG/JPEG。

#### 架构

```
┌─────────────────────────────────────────────┐
│          前端渲染管线 (Web Worker)              │
│                                              │
│  Markdown ─→ 解析 AST ─→ Canvas 布局 ─→ 分页   │
│                     ↑                        │
│               Pretext measure                │
│              (prepare + layout)              │
│                                              │
│              OffscreenCanvas                 │
│                                              │
│  ImageData ─→ Blob ─→ download / upload      │
└─────────────────────────────────────────────┘
```

#### 技术要点

1. **Markdown 解析**：在 Worker 中运行 `marked` 或 `remark` 解析 AST
2. **Canvas 布局**：遍历 AST，对每个段落/标题/列表项：
   - 用 Pretext 测量文本宽度
   - 根据可用宽度计算换行
   - 用 Canvas `fillText` 渲染
3. **分页**：当内容超过画布高度时，分割到下一页
4. **CJK 支持**：通过 Canvas system-ui 字体原生支持
5. **异步 Worker**：通过 Web Worker 避免阻塞主线程

#### 实现步骤

1. **Step 1**: 创建 `web/src/export/renderer.ts` — 核心 Canvas 渲染函数（单个段落布局）
2. **Step 2**: 创建 `web/src/export/markdown-to-canvas.ts` — Markdown AST → Canvas 布局转换
3. **Step 3**: 创建 `web/src/export/export.worker.ts` — Web Worker 封装
4. **Step 4**: 创建 `web/src/export/download.ts` — Blob 下载 / 上传接口
5. **Step 5**: 集成到前端导出 UI（替换当前 stub toast）

#### 涉及文件

- `web/src/export/` — 新建目录，包含上述文件
- `web/package.json` — 新增 `marked` 或 `remark` 依赖（可选）
- `web/src/features/query-space/pages/QueryAnalysisChat.tsx` — 导出按钮连线

### 2.2 后端 PDF CJK 支持

#### 当前瓶颈

后端现有 PDF 导出使用 `fpdf.FPDF` + Helvetica 字体，不支持 CJK 字符：

```python
# core/helpers/graph_helpers/content_support/export/drivers.py
pdf = FPDF()
pdf.add_font("Helvetica", "", "/path/to/helvetica.ttf", uni=True)  # Latin only
```

CJK 字符在输出中被替换为占位符（`□`）。

#### 方案对比

| 方案 | 原理 | CJK 支持 | 依赖复杂度 | 性能 | 推荐 |
|------|------|----------|-----------|------|------|
| **A: weasyprint** | HTML → CSS → PDF（WebKit 渲染） | ✅ 原生（加载 CJK web 字体） | 中（系统 C 库） | 中 | ⭐ 推荐 |
| **B: playwright** | Headless Chromium 截图/PDF | ✅ 原生 | 高（~400MB browser） | 低 | 备选 |
| **C: fpdf2 + CJK** | fpdf2 + NotoSansSC 嵌入 | ✅ 需嵌入字体文件 | 低 | 高 | 轻量替代 |

#### 推荐方案：weasyprint

**理由**：
- 与现有 HTML 管线无缝衔接——后端已有 `markdown_to_html_document()` 生成 HTML
- 原生支持 CSS `@font-face` 加载 CJK web 字体
- 比 playwright 轻量（无 browser 依赖）
- 输出质量高（保留 CSS 样式、表格布局、分页控制）

**实施步骤**：

1. **安装**：`pip install weasyprint` + 系统字体配置（`NotoSansSC`）
2. **创建渲染器**：在 `core/helpers/graph_helpers/content_support/export/` 新增 `weasyprint_renderer.py`
3. **HTML 增强**：在 `html_render.py` 的 HTML 文档中注入 CJK 字体 `@font-face` 和分页 CSS
4. **驱动注册**：在 `drivers.py` 中注册 `'pdf'` 格式的 weasyprint 处理函数
5. **回退机制**：weasyprint 不可用时回退 fpdf

#### 涉及文件

- `core/helpers/graph_helpers/content_support/export/drivers.py` — 修改
- `core/helpers/graph_helpers/content_support/export/weasyprint_renderer.py` — **新建**
- `core/helpers/graph_helpers/content_support/html_render.py` — 修改（字体注入）
- `pyproject.toml` 或 `requirements.txt` — 新增 weasyprint 依赖

### 2.3 看板截图导出

#### 挑战

看板通过 iframe 嵌入（`AskDataDashboardCard`），内容为 Plotly Dash 应用。前端无法直接截图跨域 iframe 内容。

#### 方案对比

| 方案 | 原理 | 跨域处理 | 适用场景 |
|------|------|----------|----------|
| **A: Dash 原生导出** | Dash 应用自身提供 `/download.png` 端点 | 无跨域问题 | 推荐（Dash 应用可自包含导出） |
| **B: 服务端 Playwright** | 后端用 Playwright 打开看板 URL 截图 | 后端无跨域限制 | 通用方案（所有看板通用） |
| **C: 前端 html2canvas** | 仅截取 iframe 外部容器（不捕获 Dash 内容） | 受限 | 仅用于占位截图 |

#### 推荐方案：方案 A + B 组合

1. **Dash 应用自包含导出**（P3.2 配合）：Dash 应用通过 `dcc.Download` 组件提供图片导出
2. **服务端回退**（P3.3）：后端 Playwright 截图作为通用回退方案

#### 涉及文件

- Dash 应用模板侧：`assets/dashboard/base_template.py` — 新增导出按钮
- 后端截图服务：`core/server/export/screenshot.py` — **新建**（Playwright 封装）
