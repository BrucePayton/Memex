## 问题描述

部署后仅创建 48 张表（含 alembic_version 和 4 个 LangGraph checkpoint 表），ORM 中定义了 60 个 artifact（58 个模型类 + 2 个关联表）。43 个业务表存在，**17 个表缺失**——均为 Claw-Code 集成新增的 MCP、Wiki、LLM 配置相关模型。[^src-1]

## 根因分析

当前部署走 **LEGACY 路径**（`RESET_ALL_LEGACY_MIGRATE=1`），复用已有 revision `80b9742e0f30_reset_all_migrate.py`（创建于 2026-04-28），该 revision 只包含 43 张表。Claw-Code 集成的 17 个新模型是在此之后添加到 ORM 的，但从未生成新的迁移 revision。`alembic upgrade head` 为 no-op，新表从未创建。[^src-1]

## 解决方案

按项目 ADR 政策：`修改 ORM → alembic revision --autogenerate → 审阅 → 提交`。

生成的新 revision `0a2119b5be45` 继承自 `80b9742e0f30`，包含：[^src-1]

- **CREATE TABLE**：17 张新表，按 FK 依赖拓扑排序
- **ALTER TABLE ADD COLUMN**：t_chain_template（8 列）、t_datasource（domain_codes）、t_external_resource_registration（platform_registration_id, external_id）、t_knowledge（domain_codes, wiki_page_count）、t_tag（domain_code）、t_workspace（domain_codes）
- **NEW FK**：t_external_resource_registration → t_platform_registration、t_tag → t_domain_config

### 审阅中排除的操作

- LangGraph checkpoint 表 DROP（运行时管理，不由 Alembic 干预）
- 已有 FK 约束的无意义重建（autogenerate 误判命名差异）

## 执行命令摘要

```bash
# 生成新 revision
alembic revision --autogenerate -m "add_missing_17_claw_wiki_llm_tables"

# 审阅后应用
alembic upgrade head  # → 新 head: 0a2119b5be45

# 验证：60 张业务表
psql ... -c "SELECT count(*) FROM pg_tables WHERE schemaname='public' ..."  # → 60

# 提交
git add storage/data_migration/alembic_data_migration/alembic/versions/0a2119b5be45_*.py
git commit -m "migration: add 17 missing tables for Claw-Code integration"
```

## 经验教训

1. **模型添加后须立即生成迁移 revision**：ORM 变更后应运行 `alembic revision --autogenerate` 并提交生成的 revision，避免 LEGACY 模式下库结构落后于 ORM
2. **Autogenerate 审阅必须严格**：autogenerate 会生成多余操作（DROP checkpoint 表、FK 重建），须逐项排除
3. **绿场 vs LEGACY**：绿场模式自动 regenerated base revision（包含全部 60 张表），问题只在 LEGACY 路径暴露
4. **LangGraph 表隔离**：checkpoint_* 表由 LangGraph 运行时管理，Alembic 应完全忽略，避免误操作
