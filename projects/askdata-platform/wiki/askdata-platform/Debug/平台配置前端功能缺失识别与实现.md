# 平台配置前端功能缺失识别与实现

## 问题描述
在开发过程中，发现外部资源管理模块存在大量前端功能缺失，需要通过历史提交记录确认缺失情况，并验证后端功能是否具备。

## 问题分析流程

### 1. 现状确认
- **后端**: 已完整实现 LLM 配置管理 (`/api/config/llm`) 和 Claw Code 管理 (`/api/claw`)
- **前端**: ROUTES 常量已定义，但缺少对应页面组件实现
- **路由**: 路由配置中未挂载这些路由

### 2. 缺失功能识别
通过对比路由定义与实际文件结构，识别出以下缺失：
- 平台配置统一入口页面 (`/platform-config`)
- LLM 配置管理页面 (`/llm-config`)
- Claw Code 管理页面 (`/claw`)
- 相关 API 封装和类型定义

## 解决方案

### 1. 后端功能验证
验证了以下后端模块已完整实现：
- `core/modules/config/views/llm_config_views.py` - LLM 配置 API
- `core/modules/claw/views/claw_views.py` - Claw Code API
- 数据库模型完整，包含 ExternalResourceRegistration 和 WorkspaceResourceMount

### 2. 前端实现方案

#### API 层封装
- `web/src/apis/config/types.ts` - LLM 配置 TypeScript 类型定义
- `web/src/apis/config/llm-config.api.ts` - LLM 配置 API 客户端
- `web/src/apis/claw/types.ts` - Claw Code TypeScript 类型定义
- `web/src/apis/claw/claw.api.ts` - Claw Code API 客户端

#### 页面组件
- `web/src/features/platform-config/pages/PlatformConfigPage.tsx` - 平台配置统一入口
- `web/src/features/llm-config/pages/LlmConfigPage.tsx` - LLM 配置管理
- `web/src/features/claw/pages/ClawDashboardPage.tsx` - Claw 会话列表
- `web/src/features/claw/pages/ClawTaskDetailPage.tsx` - Claw 任务详情

#### 路由与权限
- 更新 `web/src/app/routes/index.tsx` 添加路由
- 更新 `web/src/app/routes/appRouteMaps.ts` 添加 pageId 映射
- 更新 `web/src/app/routes/routeAccess.ts` 添加权限检查
- 更新 `web/src/features/home/pages/HomePage.tsx` 添加平台配置入口卡片

## 实现要点

### 权限控制设计
遵循现有权限模式，通过 Keycloak 中的 page_group 控制访问：
- `platform-config` - 平台配置主入口
- `llm-config` - LLM 配置管理
- `claw` - Claw Code 管理

### UI/UX 设计
- 平台配置页面作为统一入口，提供各功能模块的导航
- LLM 配置页面采用 Tab 布局，包含 Provider 管理、配置版本、配置历史
- Claw 页面采用会话列表 + 详情的双层布局
- 所有页面遵循现有设计风格，使用 Tailwind CSS 和 shadcn/ui

## 经验总结
1. **先验证后端**: 在实现前端前，先确认后端功能是否完整可用
2. **遵循现有模式**: 权限控制、路由配置、组件结构都应遵循现有代码模式
3. **统一入口设计**: 对于平台级配置，采用统一入口 + 子模块导航的模式更清晰
4. **权限优先**: 在页面实现前先完成路由和权限配置，确保安全性
