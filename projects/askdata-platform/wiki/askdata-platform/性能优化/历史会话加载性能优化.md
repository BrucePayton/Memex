# 历史会话加载性能优化

## 问题背景

在Askdata Platform中，加载历史会话时出现严重的性能问题：

- **现象**：用户打开历史会话时，页面出现卡死
- **根因**：虽然只有几条原始消息，但每条消息可能包含几十个阶段（orchestrator_trace）
- **数据规模**：测试发现一个典型会话有4条原始消息，但展开后有24个阶段消息

## 问题分析

### 技术根因

1. **阶段消息膨胀**：
   - 每条assistant消息可能包含大量的orchestrator_trace
   - 每个阶段都被渲染为独立的Collapsible组件
   - 即使有懒渲染，Collapsible组件本身仍会被创建

2. **初始渲染压力**：
   - 大量DOM节点同时渲染
   - React状态更新频繁
   - 浏览器主线程阻塞

3. **缺少限制机制**：
   - 历史加载时没有阶段数量限制
   - 所有阶段默认展开
   - 没有分页或虚拟化

## 优化方案

### 1. 阶段消息数量限制

**实现**：
- 默认只显示最后N个阶段（N=5）
- 更早的阶段被隐藏
- 添加"显示更多阶段"按钮

**代码示例**：
```typescript
const MAX_STAGES_TO_SHOW = 5; // 历史会话默认最多显示5个阶段
const stagesToProcess = traceArr.length > MAX_STAGES_TO_SHOW
  ? traceArr.slice(-MAX_STAGES_TO_SHOW) // 只保留最后N个阶段
  : traceArr;
```

### 2. "显示更多"按钮

**功能**：
- 当阶段超过限制时，显示友好的提示按钮
- 用户点击后可展开查看所有阶段
- 支持再次点击收起

**实现细节**：
```typescript
if (traceArr.length > MAX_STAGES_TO_SHOW) {
  result.push({
    id: `hist-stage-more-${row.turn_index}`,
    content: `还有 ${traceArr.length - MAX_STAGES_TO_SHOW} 个早期阶段已隐藏`,
    // ...
    isShowMoreButton: true,
    allStages: traceArr, // 保存所有阶段数据以便展开
  });
}
```

### 3. 默认折叠状态

**策略**：
- 所有历史阶段消息默认设置为折叠
- 所有阶段组也默认设置为折叠
- 用户需要手动点击展开才能看到内容

**代码**：
```typescript
// 将所有历史阶段消息默认设置为折叠状态
const collapsedState: Record&lt;string, boolean&gt; = {};
for (const msg of withSeqs) {
  if (msg.type === 'orchestrator-stage' &amp;&amp; !msg.isShowMoreButton) {
    collapsedState[msg.id] = false;
  }
}
setStageCardOpen(collapsedState);
```

### 4. 懒渲染优化

**原则**：
- 只有展开的阶段才会渲染详细内容
- 子阶段也只在父阶段展开时才渲染
- 减少初始DOM节点数量

## 实现效果

### 性能提升

- **初始渲染**：减少80%+的DOM节点
- **加载速度**：从秒级降到毫秒级
- **用户体验**：不会再出现页面卡死

### 用户体验

1. **渐进式加载**：
   - 只显示最重要的最新阶段
   - 用户按需展开查看更多

2. **可发现性**：
   - 明确提示有隐藏内容
   - 一键展开查看全部

3. **控制权**：
   - 用户完全掌控查看的内容量
   - 可随时收起以保持界面整洁

## 后续优化建议

### 短期优化
1. **虚拟列表**：对于极长的消息列表使用react-window等虚拟化库
2. **骨架屏**：历史加载时显示更友好的骨架屏
3. **加载进度**：大型会话显示加载进度

### 中期优化
1. **后端分页**：历史消息分页加载，每次只加载一页
2. **阶段过滤**：默认只显示重要阶段（如major_phase）
3. **会话摘要**：先显示会话摘要，用户确认后再加载详情

### 长期优化
1. **数据压缩**：对历史阶段数据进行压缩存储
2. **智能缓存**：根据用户习惯预加载可能访问的会话
3. **离线优先**：支持已加载会话的离线访问

## 最佳实践

### 何时使用此模式

1. **大量子项**：当一条记录可能展开为大量子记录时
2. **历史数据**：历史记录通常不需要一次性查看全部
3. **时间敏感**：用户关注最新的内容，较早的内容可以延迟加载

### 设计原则

1. **渐进式披露**：先显示最重要的内容
2. **用户控制**：让用户决定查看多少内容
3. **明确反馈**：清楚告知有隐藏内容
