# Domain Config功能规范
## 1. 概述
Domain Config是一套面向多团队协作的企业级配置管理体系，解决传统配置管理中存在的：
- 配置分散、不统一
- 变更不可追溯
- 敏感信息泄露风险
- 变更生效不灵活
等问题，实现配置的全生命周期管理。
## 2. 核心架构
### 2.1 分层存储模型
配置分为四个层级，优先级从低到高：
| 层级 | 作用域 | 优先级 |
|------|--------|--------|
| 全局配置 | 全平台 | 1 |
| 业务域配置 | 特定业务域 | 2 |
| 项目配置 | 单个项目 | 3 |
| 环境配置 | 特定环境（dev/test/prod） | 4 |
### 2.2 系统架构
```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│  配置API层  │───▶│  权限管控层  │───▶│  存储引擎层  │
└─────────────┘    └─────────────┘    └─────────────┘
       ▲                  ▲                  ▲
       │                  │                  │
┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│  控制台页面  │    │  审计日志层  │    │  缓存加速层  │
└─────────────┘    └─────────────┘    └─────────────┘
```
## 3. 核心功能
### 3.1 配置管理
- ✅ **多类型支持**：文本、数值、布尔、JSON、敏感信息、文件
- ✅ **版本管理**：全量版本历史，支持一键回滚
- ✅ **环境隔离**：不同环境配置完全隔离，支持一键同步
- ✅ **动态生效**：配置变更无需重启应用，实时生效
### 3.2 权限管控
- ✅ **RBAC权限模型**：基于角色的访问控制
- ✅ **审批流程**：重要配置变更需多人审批
- ✅ **敏感信息加密**：密钥、密码等敏感信息自动加密存储
- ✅ **灰度发布**：配置变更支持按比例、按应用灰度发布
### 3.3 可观测性
- ✅ **全链路审计**：所有配置变更操作全量记录
- ✅ **生效监控**：配置变更后自动监控生效范围
- ✅ **依赖图谱**：自动分析配置与应用的依赖关系
- ✅ **异常告警**：配置变更导致异常时自动回滚并告警
## 4. API规范
### 4.1 读取配置
```http
GET /api/domain-config/{domain}/{key}
Header: X-API-Key: {app-key}
```
响应：
```json
{
  "key": "db.connection",
  "value": "mysql://xxx:3306/askdata",
  "type": "text",
  "version": "v1.2.0",
  "lastModified": "2025-05-04T12:00:00Z",
  "description": "AskData生产库连接地址"
}
```
### 4.2 写入配置
```http
POST /api/domain-config/{domain}
Content-Type: application/json
{
  "key": "db.connection",
  "value": "mysql://xxx:3306/askdata",
  "type": "text",
  "description": "AskData生产库连接地址",
  "applyStrategy": "grayscale",
  "grayscalePercentage": 30
}
```
## 5. 最佳实践
### 5.1 配置命名规范
遵循`{业务域}.{项目}.{环境}.{功能}.{配置项}`格式：
```
payment.order.prod.db.connection
user.profile.dev.sms.expire-time
```
### 5.2 敏感信息管理
所有密钥、密码等敏感信息必须使用`SECRET`类型存储，禁止明文写入代码或配置文件。
### 5.3 变更流程
生产环境配置变更必须遵循：
1. 测试环境验证
2. 提交变更申请
3. 双人审批
4. 灰度发布（30%流量运行30分钟）
5. 全量生效
