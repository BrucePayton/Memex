# Askdata 本地部署方案最佳实践与数据库迁移经验

## 一、当前部署方案核心机制（已验证正确）

### 1.1 自动迁移执行流程
`deploy-local.sh` 部署时会自动启动 `data-migration` 服务，作为 `askdata_backend` 的前置依赖：
- 每次部署都会执行 `alembic upgrade head` 应用所有新增迁移脚本
- 即使存在 `migration.lock` 文件，仍会执行结构迁移（仅跳过种子数据初始化）
- 本地新增的迁移脚本会通过仓库挂载自动生效，无需重新构建镜像

### 1.2 数据模式说明
默认使用 `ASKDATA_DB_INIT_MODE=preserve` 保留数据模式：
- ✅ 不会删除现有业务数据
- ✅ 仅执行增量表结构变更
- ✅ 安全适用于开发和生产环境
- ⚠️ 如需清空重建，需显式设置 `ALLOW_DESTRUCTIVE_DB_INIT=1` 并切换到 `greenfield` 模式

### 1.3 挂载配置
`data-migration` 服务会挂载整个仓库根目录到 `/app`：
- 迁移脚本位置：`/app/storage/data_migration/alembic_data_migration/alembic/versions/`
- 所有本地新增的 `*.py` 迁移脚本会自动被检测到并执行

## 二、数据库迁移常见问题处理

### 2.1 手动执行SQL后的迁移标记
如果已经手动在数据库中执行了SQL变更，需要标记对应迁移为已应用，避免下次部署重复执行导致错误：
```bash
# 进入后端容器标记迁移版本
./deploy-local.sh exec askdata_backend python -m alembic -c storage/data_migration/alembic_data_migration/alembic.ini stamp <迁移版本号>

# 示例：标记本次wiki字段迁移为已应用
./deploy-local.sh exec askdata_backend python -m alembic -c storage/data_migration/alembic_data_migration/alembic.ini stamp 73036f002495
```

### 2.2 迁移状态验证命令
```bash
# 查看当前数据库已应用的迁移版本
./deploy-local.sh exec askdata_backend python -m alembic -c storage/data_migration/alembic_data_migration/alembic.ini current

# 查看所有可用的迁移版本
./deploy-local.sh exec askdata_backend python -m alembic -c storage/data_migration/alembic_data_migration/alembic.ini heads

# 检查待执行的迁移
./deploy-local.sh exec askdata_backend python -m alembic -c storage/data_migration/alembic_data_migration/alembic.ini check
```

### 2.3 迁移失败处理
如果迁移执行失败：
1. 先备份数据库
2. 查看迁移日志：`./deploy-local.sh logs data-migration`
3. 手动修复问题后，重新执行迁移：`./deploy-local.sh postgres-migration`

## 三、生产环境优化建议

### 3.1 镜像构建优化
当前本地开发模式通过挂载目录使用迁移脚本，生产环境建议：
- 将迁移脚本直接构建到 `data-migration` 镜像中，避免依赖外部挂载
- 镜像中包含所有版本的迁移脚本，确保可追溯和回滚

### 3.2 CI/CD流程优化
在流水线中添加：
- 迁移脚本语法校验：`alembic check`
- 兼容性检查：验证迁移不会破坏现有数据
- 预演执行：在测试环境先执行迁移，验证无问题后再上线

### 3.3 回滚预案
重要变更前：
1. 完整备份数据库
2. 提前准备好回滚脚本：`alembic downgrade -1`
3. 先在灰度环境验证，再全量上线
