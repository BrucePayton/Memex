## 一、当前项目架构概览

### 系统组件

| 组件 | 职责 | 状态 |
|------|------|------|
| `dashboard/server.py` | HTTP API + UI 服务 | ✅ 核心可用 |
| `dashboard/index.html` | 单页应用 UI | ✅ 核心可用 |
| `mcp-server/memex_mcp.py` | MCP 工具暴露 | ✅ 基本可用 |
| `project_registry.py` | 多项目管理 | ✅ 可用 |
| `index_strategy.py` | 索引策略（flat→hierarchical→indexed）| ✅ 可用 |
| `provenance.py` | 来源图谱 | ✅ 可用 |

### 工作流

```
用户添加 source (raw/)
    ↓
[Ingest] → Claude 处理
    ↓
生成 wiki/ 页面 (source-summary + entities + concepts)
    ↓
git 自动提交
    ↓
[Query] / [Lint] / [Reflect] / [Compare] / [Write]
```

## 二、功能缺陷清单

### P0 - 阻断性问题

| ID | 问题 | 影响 | 根因 |
|----|------|------|------|
| **DEF-001** | **Raw 文件预览不完整** | PDF/图片预览刚修复，但很多格式无预览 | `api_raw_file` 仅支持部分格式，非文本格式没有好的预览方案 |
| **DEF-002** | **中文文件名在 HTTP header 中编码问题** | 刚修复，但需要验证边界情况 | `send_header` 只支持 Latin-1，中文需要 RFC 5987 编码 |
| **DEF-003** | **Raw 文件上传缺失** | 只能通过 Ingest 粘贴文本，无法直接上传文件到 raw/ | UI 没有上传组件，API 没有上传端点 |
| **DEF-004** | **非 Markdown Raw 格式无法 Ingest** | PDF/Word/EPUB 等无法直接 Ingest | `do_ingest` 只接受纯文本内容 |

### P1 - 高优先级问题

| ID | 问题 | 影响 | 根因 |
|----|------|------|------|
| **DEF-005** | **Lint 自动化程度低** | Lint 报告后需要人工确认修复 | `do_lint` 生成报告，`do_lint_fix` 需单独触发且不完善 |
| **DEF-006** | **Contradiction 检测缺失** | 新 source 与旧 wiki claim 冲突时不会自动标记 | 只有 CLAUDE.md 规范，没有实现自动化检测 |
| **DEF-007** | **Citation 完整性检查薄弱** | 很多事实 claim 没有 citation | Lint 仅检查有无定义，不检查覆盖率 |
| **DEF-008** | **History / Git revert UI 缺失** | 误操作后难以恢复 | 有 `/api/history` 但没有恢复功能 |
| **DEF-009** | **Query 回答没有自动保存选项** | 有价值的回答可能丢失 | `do_query` 返回回答但不自动创建 analysis 页面 |
| **DEF-010** | **搜索仅限文件名和标题** | 无法搜索页面正文内容 | `do_search` 只看 frontmatter |

### P2 - 中优先级问题

| ID | 问题 | 影响 | 根因 |
|----|------|------|------|
| **DEF-011** | **无批量 Ingest 功能** | 多个 source 只能逐个处理 | 设计上就是单 source ingest |
| **DEF-012** | **无标签管理/云标签功能** | 标签分散在各页面 frontmatter | 没有统一标签视图/管理 |
| **DEF-013** | **页面无版本对比（diff）** | 看不到某页面随时间的变化 | 有 git 历史但没有 diff UI |
| **DEF-014** | **Graph 视图交互性弱** | 只能看，不能点击跳转 | D3.js 渲染但没有交互绑定 |
| **DEF-015** | **无导出功能** | 无法将 wiki 导出为其他格式 | 没有导出模块 |
| **DEF-016** | **无备份/归档功能** | 项目多了后管理困难 | 没有归档/备份流程 |

### P3 - 低优先级问题

| ID | 问题 | 影响 |
|----|------|------|
| **DEF-017** | **UI 响应式不完善** | 移动端体验不好 |
| **DEF-018** | **无深色/浅色主题切换** | 当前只有深色主题 |
| **DEF-019** | **快捷键可配置性** | 目前是硬编码的快捷键 |

## 三、Raw 文件处理的完整缺陷链

### 当前流程的断点

```
用户有 PDF 文件
    ↓
[问题 1] 无法直接上传到 raw/（DEF-003）
    ↓
[问题 2] 即使放进去了也无法 Ingest（DEF-004）
    ↓
[问题 3] 只能预览，但无法提取知识（DEF-001）
```

### 建议的 Raw 处理增强

| 格式 | 预览方案 | Ingest 方案 |
|------|----------|-------------|
| PDF | iframe 预览 + 文本提取（pdfminer.six） | OCR/文本提取 → ingest 文本 |
| Word (.docx) | mammoth.js 转换 HTML 预览 | 文本提取 → ingest |
| EPUB | epub.js 阅读器 | 文本提取 → ingest |
| 图片 | img 标签 | 可选：OCR → ingest |
| 视频 | 无法预览 | 元数据 ingest |
| 代码 | syntax highlight 预览 | 可选：ingest 作为 source |

## 四、助手定位分析

### 当前两个助手的分工

| 助手 | 位置 | 能力边界 | 当前问题 |
|------|------|----------|----------|
| **Floating Dashboard Helper** | 右下角浮动 Claude | 只回答 dashboard 使用问题，不碰 wiki 内容 | ✅ 定位清晰，问题少 |
| **External Claude (MCP)** | Claude Code/Desktop | 14 个 MCP 工具，完整读写 wiki | ⚠️ 定位重叠 |

### 定位重叠问题

**Dashboard vs MCP**：

- Dashboard: Ingest/Query/Lint/Reflect/Compare/Write 按钮
- MCP: `add_raw_source` / `create_page` / `update_page` / `search` / etc

**问题**：
1. 用户不知道该用哪个
2. 功能重复实现（Ingest 在两边都有）
3. MCP 工具不知道 Dashboard 已有的状态

### 建议的定位清晰化

#### 方案 A - "Dashboard = 方向盘；MCP = 机械手"

| 入口 | 用途 | 示例 |
|------|------|------|
| **Dashboard** | 浏览、提问、管理项目、一键式操作 | "Ingest 这个文件"、"Query X"、"Lint 并修复" |
| **MCP** | 细粒度编辑、批量操作、复杂工作流 | "基于搜索结果创建 5 个概念页面"、"批量重命名标签" |

#### 方案 B - "Dashboard = 日常使用；MCP = 高级用户"

- Dashboard 优化：让 90% 操作在 Dashboard 完成
- MCP 保留：给需要程序化/批量/复杂操作的高级用户

## 五、MCP 工具设计的问题

### 当前工具集

| 读工具 | 写工具 |
|--------|--------|
| `list_projects` | `add_raw_source` |
| `list_pages` | `create_page` |
| `read_page` | `update_page` |
| `search` | `create_folder` |
| `folder_tree` | `git_commit` |
| `stats` | |
| `recent_log` | |
| `list_raw_sources` | |
| `get_instructions` | |

### 问题

1. **缺少语义化工具**：
   - 没有 `ingest` 工具（虽然有 `add_raw_source`，但只是写文件，不触发处理）
   - 没有 `query` 工具
   - 没有 `lint` 工具

2. **工具职责不清晰**：
   - `git_commit` 需要手动调用，但 Dashboard 是自动 commit
   - MCP 写入不自动 commit，导致不一致

3. **工具缺少上下文**：
   - 每个工具都要传 `project` 参数
   - 没有"当前项目"的概念

### 建议的 MCP 增强

```python
# 新增语义化工具
@mcp.tool()
def ingest(title: str, content: str, project: str = None) -> dict:
    """Ingest a source into the wiki (triggers full ingest pipeline)."""

@mcp.tool()
def query(question: str, project: str = None) -> dict:
    """Ask a question against the wiki."""

@mcp.tool()
def lint(project: str = None) -> dict:
    """Run lint and return report."""

@mcp.tool()
def set_current_project(project: str) -> dict:
    """Set default project for subsequent tools (avoids passing project every time)."""
```

## 六、Wiki 体系的缺陷

### Citation 机制的问题

| 问题 | 说明 |
|------|------|
| **覆盖率无指标** | Lint 报告没有"citation 覆盖率"指标 |
| **citation 粒度粗** | 整句引用，无法引用段落/图表 |
| **无反向链接视图** | 某 source 被哪些页面引用？只有 provenance graph |
| **citation 验证缺失** | `[^src-xxx]` 指向的 source-summary 页面是否真的支持那个 claim？|

### 页面类型的局限性

| 类型 | 用途 | 问题 |
|------|------|------|
| `source-summary` | 单 source 摘要 | ✅ 好 |
| `entity` | 人名/机构/产品 | ✅ 好 |
| `concept` | 想法/框架 | ✅ 好 |
| `technique` | 方法/算法 | ✅ 好 |
| `analysis` | 综合分析 | ❓ 粒度太粗，没有子类型 |
| `overview` | 首页 | ✅ 好 |
| `comparison` | 对比 | ❓ 这个类型是否真的需要？|

**建议**：考虑是否需要 `comparison` 作为独立类型，或者并入 `analysis`。

### Index 策略问题

| 规模 | 策略 | 问题 |
|------|------|------|
| <50 | flat index | ✅ 好 |
| 50-200 | hierarchical | ❓ 自动分类质量如何？|
| >200 | indexed | ❓ 实现了吗？|

## 七、多项目隔离的缺陷

### 当前状态

- ✅ 项目在 `projects/<slug>/` 下完全隔离
- ✅ git commit 带 slug 前缀
- ⚠️ Dashboard 切换项目，但 MCP 需要手动传 project 参数

### 问题

1. **项目间知识无法复用**：
   - Project A 有"Python"概念，Project B 也有，无法跨项目引用
   - 没有"公共概念"的概念

2. **没有项目模板继承**：
   - 创建新项目只能从空白或简单模板开始
   - 无法从现有项目克隆/派生

3. **没有项目分组/标签**：
   - 项目多了后是平级列表
   - 无法分类/分组

## 十一、关键架构决策点

### 决策点 1 - 是否引入外部依赖？

当前状态：**0 pip dependencies**（仅 stdlib）

| 方案 | 优点 | 缺点 |
|------|------|------|
| **保持 0 依赖** | 简单、快速启动、无依赖地狱 | PDF/DOCX 解析需自己写或用 subprocess 调用外部工具 |
| **引入可选依赖** | PDF 解析用 `pdfminer.six`、DOCX 用 `python-docx` | 用户需安装；版本问题 |
| **混合方案** | 基础功能 0 依赖，高级功能检查可选依赖 | 实现复杂 |

**建议**：混合方案。检查依赖是否存在，不存在则降级功能。

### 决策点 2 - Dashboard vs MCP 的边界

| 操作 | 应该在 Dashboard？| 应该在 MCP？|
|------|-----------------|------------|
| Ingest 单个文件 | ✅ | ✅ |
| 批量 Ingest 100 个文件 | ❌ | ✅ |
| Query 一个问题 | ✅ | ✅ |
| "重跑过去 30 天的 Query 并更新分析" | ❌ | ✅ |
| Lint 并显示报告 | ✅ | ✅ |
| "Lint 并自动修复所有问题" | ✅ | ✅ |
| "跨项目统一标签风格" | ❌ | ✅ |

**建议**：Dashboard 做"单步、交互式"操作；MCP 做"批量、程序化"操作。

### 决策点 3 - 多项目关系

| 方案 | 说明 |
|------|------|
| **完全隔离** | 当前方案，项目间无联系 |
| **项目继承** | 项目可以继承另一个项目的 wiki |
| **公共库** | 有一个 `_common` 项目，其他项目可以引用 |
| **跨项目引用** | `[[project-slug:page-name]]` 语法 |

**建议**：先保持完全隔离，收集用户反馈后再考虑。

## 十二、总结

### 核心问题

1. **Raw 处理链不完整**（上传→解析→ingest→预览）
2. **QA 自动化程度低**（Contradiction 检测、Citation 覆盖率）
3. **助手定位有重叠**（Dashboard vs MCP）
4. **测试覆盖不足**

### Quick Wins

| 任务 | 工作量 | 收益 |
|------|--------|------|
| Raw 文件上传 | 1-2 天 | 高 |
| Git revert UI | 2-3 天 | 高 |
| 全文搜索 | 3-5 天 | 中高 |
| MCP 语义化工具 | 2-3 天 | 中 |

### 长期方向

- **专注核心**：把 wiki 维护自动化做到极致
- **清晰边界**：Dashboard = 日常使用，MCP = 高级/批量操作
- **渐进增强**：保持 0 依赖基础，高级功能可选依赖
